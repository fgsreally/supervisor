## 一、上下文管理与压缩

### 1.1 Full Compaction + 第一人称 Handoff Prompt

**来源：** kimi-code `packages/agent-core/src/agent/compaction/`

压缩不是截断，是"交接"。专用 compaction prompt 要求模型写**第一人称、可执行的接续笔记**，包含：精确命令和路径、上一步结果值、未验证假设、前进计划。压缩后上下文形态：`[保留的 HEAD 用户消息] + [第一人称 summary 消息]`，中间所有 assistant/tool 历史全部丢弃。

触发条件（默认 85% 上下文 + 50k 预留输出空间）：

```
triggerRatio: 0.85
reservedContextSize: 50_000
maxOverflowCompactionAttempts: 3
```

**system prompt 同步教育模型**：专门有一节说明"收到 summary 后如何接续工作"，避免模型因失忆而重复已完成的工作。

**与 supervisor 的对比：** supervisor 已有 rolling compaction，但 handoff prompt 的质量和 system prompt 里的配套教育不明确。

---

### 1.2 多层上下文协作管理（Prune → Compaction → Checkpoint Rebuild → Microcompact）

**来源：** MiMo-Code `session/prompt.ts`、`session/checkpoint.ts`

不是单一压缩策略，而是四层协作：

1. **Prune**（机械擦除）：从旧消息向后遍历，将超出 40K token 保护边界的旧 tool result 内容擦除，skill 工具输出受保护。
2. **Compaction**（LLM 摘要）：专用 hidden agent，`toolAllowlist: []`，按 turn 保留最近 N 轮 + token budget 选择分割点。
3. **Checkpoint + Rebuild**（语义保留重建）：接近上限时 fork `checkpoint-writer` 子代理写结构化快照，overflow 时从 checkpoint + memory + 最近消息重建上下文，而非盲目截断。
4. **Microcompact**（tail 区占位）：对 rebuild boundary 之后的可再生工具结果（read/bash/grep）替换为占位符，保留 tool_use 结构体。

子代理 overflow 和主代理 overflow 走不同路径：子代理触发 per-actor compaction，主代理触发 checkpoint rebuild。

---

### 1.3 结构化 11 节 Checkpoint + Rebuild Boundary

**来源：** MiMo-Code `session/checkpoint.ts`、`agent/prompt/checkpoint-writer.txt`

接近上下文上限时，后台 fork 专用 `checkpoint-writer` 子代理（继承父代理 LLM prefix cache），写入结构化 `checkpoint.md`，包含 11 个章节：

1. 会话意图 / 用户原始目标
2. 当前状态（已完成 / 进行中 / 待处理）
3. 任务树（与 SQLite task DB 对齐）
4. 关键文件清单（路径 + 最后已知状态）
5. 错误与决策日志
6. Verbatim 约束（必须原样保留的值：命令、路径、ID）
7. 环境假设
8. 前进计划
9. 待验证假设
10. 已尝试路径（避免重复）
11. 与上一 checkpoint 的变更 diff

主循环 overflow 时插入 **rebuild boundary**，从 checkpoint + memory + 最近消息重建，不丢连续性。

**Verbatim 约束** 是关键细节：某些精确值（token、ID、命令）不能被摘要改写，checkpoint 必须原样保留。

---

### 1.4 Token Budget 控制记忆注入量

**来源：** MiMo-Code `session/budgeted-read.ts`

`MEMORY.md`、`checkpoint.md`、`notes.md`、`tasks/<id>/progress.md` 等记忆文件不是全量注入，而是通过 token budget 控制注入量。避免记忆文件膨胀后撑爆 system prompt。

---

### 1.5 Auto Compact + SummaryMessageID 截断

**来源：** opencode `internal/tui/tui.go`、`internal/llm/agent/agent.go`

TUI 在每次 agent_end 后检查 token 使用率，超过 95% 自动触发压缩。压缩完成后通过 `SummaryMessageID` 字段标记摘要位置，下次构建 messages 时从摘要消息开始，之前历史全部丢弃。summary 消息的 role 改为 `user`，使上下文形态合法。

---

### 1.6 Prompt Origin 分类驱动上下文语义

**来源：** kimi-code `agent/context/projector.ts`

每条消息带 `origin` 字段：`user / injection / compaction_summary / system_trigger / …`。压缩、渲染、保留策略都基于 origin——比纯 role-based 截断更精确。

`compaction_summary` 来源的消息在下次压缩时被识别为前一次交接，不会被再次摘要。`injection` 来源的消息在重建时可以重新注入，不需要保留原文。

---

### 1.7 边界注入管理器（InjectionManager）

**来源：** kimi-code `agent/injection/`

Goal、ToolsDiff、PlanMode 提醒等不是每步注入，而是在 turn 边界 **append-only** 注入，避免 per-step 注入导致 prompt cache 失效和 O(n²) 上下文增长。

内置注入器：

| 注入器                     | 触发时机              |
| -------------------------- | --------------------- |
| GoalInjector               | 边界，保 prompt cache |
| ToolsDiffInjector          | 可加载工具变更公告    |
| PlanModeInjector           | 进入/退出规划模式     |
| PermissionModeInjector     | 权限模式变更          |
| TodoListReminderInjector   | 长时间未更新 todo     |
| PluginSessionStartInjector | 插件会话启动          |

---

### 1.8 消息投影与 Wire 修复

**来源：** kimi-code `agent/context/projector.ts`

自动修复 Provider 要求的消息格式约束：tool_use/tool_result 必须邻接、连续 assistant 合并、合成缺失 result。

严格重发模式（`buildMessagesStrict`）：Provider 因消息格式返回 400 时，一次性用严格模式重建消息重发，记录 anomaly 但不静默。

## 二、工具系统

### 2.1 LSP 诊断自动注入工具结果

**来源：** opencode `internal/llm/tools/edit.go`、`internal/app/lsp.go`

每次 `edit` / `write` / `patch` / `view` 执行后，**自动** 等待 LSP 诊断（最多 5s），并将结果追加到工具返回值：

```
<file_diagnostics>    当前文件的错误/警告
<project_diagnostics> 项目级全量诊断
<diagnostic_summary>  摘要
```

agent 无需主动调用 `diagnostics` 工具，每次写操作后立刻看到自己引入的 type error，形成修改→诊断→修复闭环。

**LSP 基础设施**：

- 每个 LSP server 独立 goroutine 启动，30s 初始化超时
- `WorkspaceWatcher`（fsnotify + 300ms debounce）同步文件变更
- 崩溃后自动 `restartLSPClient`

**与 supervisor 的对比：** supervisor 的 `lsp` 是独立打包扩展，需 agent 主动调用，不会自动注入 edit/write 结果。

---

### 2.2 先读后改强制校验 + 修改时间戳验证

**来源：** opencode `internal/llm/tools/file.go`、`internal/llm/tools/patch.go`

全局 `fileRecords` map 追踪每个文件的最后 read 时间和 write 时间。edit/patch 执行前：

1. 没有 read 记录 → 返回错误：`you must read the file before editing it`
2. 文件在 read 之后被其他工具修改 → 返回错误：`file has been modified since it was last read`

防止 agent 凭幻觉或过期记忆直接 patch，对多 agent 并发场景尤为重要。

---

### 2.3 工具调用去重 / Doom Loop 检测

**来源：** kimi-code `agent/turn/tool-dedup.ts`；MiMo-Code `session/processor.ts`

**kimi-code 方案**：同工具同参数重复时，在工具结果中追加 `<system-reminder>`，逐级升级：第 3/5/8/12 次提醒，12 次后强制停止工具调用。

**MiMo-Code 方案**：连续 3 次相同 tool+input 签名（`stableStringify` 比较）→ 触发 `doom_loop` 权限询问，让用户决定是否继续。

两者互补：kimi-code 的方案对轻度重复更温和，MiMo-Code 的方案对严重死循环有强制中断。

---

### 2.4 工具输出 Token 预算 + 落盘分页

**来源：** kimi-code `tools/builtin/`

超过 50k 字符的工具结果落盘，模型只看到 preview + `output_path`，用 Read 工具分页读取。相比 supervisor 的 output-minimizer（事后过滤），这是从源头控制。

opencode 也有硬限制：bash 输出 30K 字符，view 工具 250KB / 2000 行。

---

### 2.5 渐进式 MCP 工具披露（Progressive Tool Disclosure）

**来源：** kimi-code `tools/builtin/select-tools.ts`、`agent/injection/tools-diff.ts`

MCP 工具 schema **不**全量塞进顶层 `tools[]`，解决 MCP 工具过多撑爆 context 的问题：

1. 通过 `<tools_added>/<tools_removed>` 公告可用工具名称
2. 模型调用 `select_tools` 按名加载具体 schema
3. Schema 以 `role: system` + `tools` 消息注入上下文
4. 下一步即可调用（`buildTools` 每 step 重建工具表）

---

### 2.6 资源冲突感知并发调度（ToolScheduler）

**来源：** kimi-code `loop/tool-scheduler.ts`、`loop/tool-access.ts`

工具声明资源访问范围，调度器基于冲突决定并行/串行：

| 访问类型                              | 说明                          |
| ------------------------------------- | ----------------------------- |
| `none()`                              | 无本地资源占用，自由并发      |
| `readFile(path)` / `readTree(path)`   | 只读，无冲突可并行            |
| `writeFile(path)` / `writeTree(path)` | 写操作，与其他写同路径串行    |
| `all()`                               | 全局互斥（如 `select_tools`） |

调度规则由代码执行，不由 LLM 决定。LLM 输出一批 tool calls 后，调度器按原始顺序检查资源冲突：不冲突的立即并发，结果仍按原始 tool call 顺序回填。

---

### 2.7 双工具调用风格（JSON / Shell Script）

**来源：** MiMo-Code `tool/invocation-style.ts`、`tool/shell-wrap.ts`

工具支持两种调用风格：JSON 参数和 Shell 脚本。Shell 模式对某些模型（如 Claude）更自然，可解析自由格式 shell 命令并映射到工具参数。

```typescript
shell?: {
  description: string
  parse(script: string): Effect.Effect<Params[], unknown>
  recover?(rawArgs: unknown): Params | undefined
}
```

---

### 2.8 工具描述即文档（超详细 Tool Description）

**来源：** opencode `internal/llm/tools/grep.go`、`view.go`、`bash.go`

每个工具的 description 包含数百字，结构为：

- WHEN TO USE（与其他工具的使用边界）
- HOW TO USE（最佳实践）
- LIMITATIONS（硬性限制）
- TIPS（提效技巧）

减少模型误用工具的频率，相当于把工具文档嵌进模型的决策链路。

---

### 2.9 Grep 按修改时间倒序排列

**来源：** opencode `internal/llm/tools/grep.go`

搜索结果按**文件修改时间倒序**排列，优先展示最近活跃文件。对 coding agent 很实用，通常最新修改的文件和当前任务最相关。

---

### 2.10 Bash 输出清洗管道

**来源：** MiMo-Code `bash_token_efficient_pipeline.ts`

对 bash 输出做 token 效率优化：

- 剥离 ANSI 转义码
- 折叠进度条/Loading 动画
- 脱敏密钥（API key 模式匹配）
- 截断超长单行
- 按命令形状选用启发式插件（git diff、pytest 结果、tsc 输出等）
- never-worse 保证：清洗后不比原文长

---

### 2.11 持久 Shell（Persistent Shell）

**来源：** opencode `internal/llm/tools/shell.go`

单例持久 shell，命令间保持环境变量和 cwd 状态。避免每次 bash 调用都是全新环境，允许 `source .env`、`cd project && ...` 等模式。

---

### 2.12 Bash 安全策略（白名单/黑名单）

**来源：** opencode `internal/llm/tools/bash.go`

- 危险网络命令（curl/wget）列入黑名单，防 prompt injection 外泄
- git status/log/diff 等只读命令免权限审批
- 强制用专用工具做文件搜索/读取，禁止 bash 中 grep/find/cat

## 三、Agent 规划与任务追踪

### 3.1 Task/Goal 双闸门防过早停止

**来源：** MiMo-Code `task/gate.ts`、`session/goal.ts`

**Task Gate**：agent 准备停止前，检查 SQLite task DB 中有无未完成任务。有则注入 `<system-reminder>` 要求对每个任务执行 `task done` 或 `task abandon`，最多重试 3 次。

**Goal Gate**：用户通过 `/goal` 设定的停止条件，由独立裁判模型（temperature=0）冷评估。裁判只看 transcript 证据，不能看到模型内部状态。`fail-open` 设计：裁判失败时允许停止，不卡死用户。最多 12 次评估循环。

两者分工：Task Gate 管计划完整性，Goal Gate 管用户定义的完成语义。

---

### 3.2 树形任务追踪（SQLite Task DB）

**来源：** MiMo-Code `task/registry.ts`、`tool/task.ts`

任务使用层级 ID（`T1`、`T1.1`、`T1.1.2`），存 SQLite。task 工具支持：创建、更新状态、标记完成、标记放弃。

与 checkpoint §3 联动：checkpoint 写入时读取 task DB 当前状态，保证快照与任务树一致。

---

### 3.3 Plan Mode（一等公民规划模式）

**来源：** kimi-code `agent/plan/`、`tools/builtin/planning/`；MiMo-Code `agent/agent.ts`

不是 prompt 里说一句"先规划"，而是工具 + 权限策略 + plan 文件形成闭环：

- `EnterPlanMode` → 创建 plan id，切换权限策略（禁止写项目文件）
- 模型只能探索 + 写 plan markdown 文件
- `ExitPlanMode` → 读取 plan 文件展示给用户审批，可携带 2-3 个可选方案
- 用户审批后恢复普通权限

MiMo-Code 的 `hardPermission` 设计：用数据（path glob allow/deny）而非 if-else 保证不变量。

---

### 3.4 Goal Mode（持久目标驱动）

**来源：** kimi-code `agent/goal/`

带预算的自主多轮执行：

- goal 状态机：`active` → `paused` / `blocked` / `complete`
- budget：turn 数、token 数、wall clock 时间
- 模型通过 `UpdateGoal(complete|blocked)` 自判结束
- goal continuation：turn 结束后若 goal 仍 active 且预算未耗尽，自动注入续跑提示
- `paused` 可恢复，`blocked` 说明系统/模型认为无法继续

---

### 3.5 TodoList + 动态提醒

**来源：** kimi-code `tools/builtin/state/todo-list.ts`、`agent/injection/todo-list.ts`

结构化 todo（pending/in_progress/done），与 compaction 联动（压缩后从 live store 重挂，不重复抄写到 summary）。长时间未更新时动态 injector 低频提醒模型更新或清理。

---

### 3.6 Workflow 确定性编排

**来源：** MiMo-Code `workflow/runtime.ts`

需求清晰时用确定性 JS 脚本固定 phase 序列，比纯对话更可靠：

- `phase()` / `parallel()` / `pipeline()` / `agent()` 宿主 API
- 12h 脚本超时、≤1000 agent 生命周期、默认并发 16
- journal checkpoint 支持可恢复
- 内置工作流：`compose`（TDD 闭环）、`deep-research`、`fact-check`
- git worktree 隔离并行任务

## 四、子代理编排

### 4.1 专精子代理 Profile（explore/coder/plan）

**来源：** kimi-code `profile/default/*.yaml`；MiMo-Code `agent/agent.ts`

| 类型                | 权限                             | 用途              |

### 4.3 AgentSwarm（大规模并行子代理）

### 4.4 Max Mode（Best-of-N + 裁判选优）

### 4.5 Git 上下文注入（explore 子代理）

explore 子代理启动前，自动收集 `<git-context>`（当前分支、最近 commit、status 等）注入到子代理 system prompt，帮助子代理定位代码变更范围。

## 五、错误处理与防卡死

### 5.1 多层重复检测

**来源：** MiMo-Code `session/prompt.ts`、`prompt/text-loop-recovery.ts`、`prompt/text-ngram-detection.ts`

三种检测机制：

1. **工具调用去重**：`stableStringify` 比较连续 tool call 签名，连续相同 → doom_loop 权限询问
2. **文本循环检测**：检测重复 prose 片段，注入恢复 prompt 打破循环
3. **N-gram 重复检测**：推理/文本流中的 n-gram 重复检测

比单一 max-steps 限制更精细，能区分"正常多步"和"卡死循环"。

---

### 5.2 截断续写提醒

**来源：** MiMo-Code `session/prompt.ts`

LLM 因输出长度被截断（`finish_reason=length`）时，下一轮自动注入续写 reminder，告诉模型上次输出被截断，请继续而非重新开始。

---

### 5.3 Recoverable Error vs Fatal Error 分离

**来源：** MiMo-Code `tool/recoverable.ts`

工具错误分为两类：

- **RecoverableError**：参数错误、ID 不存在等，TUI 静音显示，模型仍收到完整错误信息和纠错 guidance
- **FatalError**：权限拒绝、不可重试的系统错误，显示给用户

改善 UX 同时给模型完整的自愈信息。

---

### 5.4 Steer 缓冲（防竞态）

**来源：** kimi-code `agent/turn/index.ts`

Turn 进行中或 manual compaction 时，新用户输入进入 `steerBuffer`，压缩/turn 结束后回放，避免竞态。

---

### 5.5 LLM 重试策略

**来源：** MiMo-Code `session/retry.ts`；opencode `internal/llm/provider/openai.go`；kimi-code `loop/retry.ts`

| 特性             | 说明                                    |
| ---------------- | --------------------------------------- |
| 指数退避         | 300ms–5000ms，含随机 jitter             |
| Retry-After 尊重 | 解析 429 响应头，等待服务器指定时间     |
| SSE 超时重试     | 流式连接超时重连                        |
| 最大重试次数     | 3–8 次（各项目不同）                    |
| 可重试错误分类   | Provider 层 `isRetryableError` 函数判断 |

---

### 6.6 System Prompt 行为契约设计原则

**来源：** kimi-code `profile/default/system.md`（约 150 行）；opencode `internal/llm/prompt/coder.go`

高质量 system prompt 的通用原则（从三个项目提炼）：

1. **行动优先**：能改代码就用工具，不把展示代码当交付
2. **并行工具调用**：只读调查强烈推荐并行 Read/Grep/Glob
3. **最小改动**：bug fix 不顺带重构，不引入无关变更
4. **权限拒绝不重试**：被拒后换方案，不绕路
5. **Compaction 接续契约**：教模型如何对待 summary 继续工作，不重做已完成事项
6. **语言跟随用户**：回复用用户语言，代码/路径保持原样
7. **工具边界明确**：每个工具说清楚 WHEN NOT TO USE，避免模型混用
8. **持续执行**："keep going until the user's query is completely resolved before ending your turn"

### 12.1 Edit 多策略模糊匹配（Fuzzy Replacer 链）

**来源：** MiMo-Code `tool/edit.ts`

`old_string` 精确匹配失败时，按序尝试多种 Replacer，显著降低模型因缩进/空白差异导致的编辑失败：

| Replacer                     | 机制                                  |
| ---------------------------- | ------------------------------------- |
| SimpleReplacer               | 精确匹配                              |
| LineTrimmedReplacer          | 按行 trim 后匹配                      |

| WhitespaceNormalizedReplacer | 空白归一化                            |
| IndentationFlexibleReplacer  | 缩进弹性                              |

### 12.2 MultiEdit（单文件批量顺序编辑）

### 12.3 Apply Patch（多文件原子 Patch）

| `spawn`                      | 后台启动                              |
| `status` / `wait` / `cancel` | 生命周期管理                          |
| `send`                       | 向另一 actor 发消息（`<inbox>` 包裹） |
| `models`                     | 列出可用模型（含 vision 过滤）        |

高级参数：

- `context`: `none` / `state` / `full` — 上下文继承级别
- `output_schema`: JSON Schema 强制结构化输出
- `task_id`: 绑定 task 树，完成后验证 `tasks/<id>/progress.md`
- `actor_id` resume：继续同一子代理而非重开
- `model`: 模型组名（ultra/standard/lite）或 provider/model

Shell 风格调用 + Levenshtein 动词纠错（`actor staus` → 建议 `status`）。

### 14.5 Prefix Capture / Fork Agent Cache

**来源：** MiMo-Code `session/prefix-capture-ref.ts`、`tool/session.ts`

checkpoint-writer、fork-query 等 fork 子代理继承父 agent 的 LLM request prefix（system prompt + tools 定义），降低重复构建成本，提高 prompt cache 命中率。

### 14.6 Swarm Mode（群集模式）

**来源：** kimi-code `agent/swarm/`

三种触发：`manual`（`/swarm on` 持久）、`task`（一次性）、`tool`（AgentSwarm 入口）。

进入后注入 swarm 提醒，权限策略切换到 swarm 专属规则（`agent-swarm-exclusive-deny` 等）。`task`/`tool` 触发在 turn 结束后自动退出。

与 AgentSwarm 工具配合，支持大规模并行探索。

**来源：** MiMo-Code `worktree/`；supervisor 已有部分实现

MiMo-Code workflow compose 可对独立任务创建 git worktree 并行开发，完成后 merge。supervisor 已有 worktree 但可借鉴 MiMo-Code 的自动命名、隔离策略。

## 十六、安全、权限与路径策略（补充）

### 16.1 外部目录写入拦截（External Directory Guard）

**来源：** MiMo-Code `tool/external-directory.ts`

写操作前检查目标路径是否在 worktree 内。worktree 外路径触发 `external_directory` 权限询问。memory 子树有独立 `memory-path-guard`，避免 headless 模式下权限死锁。

### 16.3 工具级 approvalRule

**来源：** kimi-code 各 builtin tool 的 `resolveExecution()`

### 16.4 Git CWD 写操作审批

**来源：** kimi-code `agent/permission/policies/git-cwd-write-approve.ts`

在 git 仓库根目录的写操作单独审批策略，防止 agent 误改 `.git` 或根级配置文件。

### 16.5 Continue Loop on Deny（权限拒绝后继续）

**来源：** MiMo-Code `session/processor.ts`（`experimental.continue_loop_on_deny`）

工具调用被用户拒绝时，默认中断 agent 循环。开启此实验选项后，注入拒绝信息让模型换方案继续，而非直接停止。

## 二十三、第三轮深挖补充（opencode / MiMo-Code / kimi-code）

以下条目来自第二轮、第三轮源码深挖，在第十二至二十二节中仅出现在优先级表或未单独成节。

### 23.1 Patch Fuzz 评分与拒绝阈值

**来源：** opencode `internal/diff/patch.go`、`internal/llm/tools/patch.go`

`TextToPatch` 解析后为每处匹配计算 fuzz 分数：精确匹配=0，trim 空白=1/100，EOF 偏移 +10000。**fuzz > 3 拒绝应用**。比二元「匹配/不匹配」更稳健，减少误 patch 同时容忍轻微格式差异。

---

### 23.2 Permission Diff 预览（Side-by-side 语法高亮）

**来源：** opencode `internal/tui/components/dialog/permission.go`、`internal/diff/diff.go`

权限审批弹窗按工具类型渲染：bash 命令、edit/write/patch 的 **side-by-side 语法高亮 diff**（chroma + udiff）。三选项：Allow / Allow for session / Deny。supervisor Web UI 的 ask/edit approval 可对标此体验。

---

### 23.3 子代理费用回卷到父 Session

**来源：** opencode `internal/llm/agent/agent-tool.go`

子 session 的 token 费用累加到 `parentSession.Cost`，统一账单视图。supervisor 子 session 已有独立记录，缺父子聚合展示。

---

### 23.4 PermissionDenied 级联取消后续 Tool Call

**来源：** opencode `internal/llm/agent/agent.go`

用户拒绝某 tool call 权限时，设置 `FinishReasonPermissionDenied`，**同轮后续 tool call 全部标记 canceled**，避免在已拒绝前提下继续执行。

---

### 23.5 Session Busy 锁

**来源：** opencode `internal/llm/agent/agent.go`

`activeRequests sync.Map` + `ErrSessionBusy`：同 session 同时只处理一个请求；summarize 用 `{id}-summarize` 独立键，防并发 prompt 竞态。

### 23.6 Write 相同内容拒绝（No-op 拦截）

**来源：** opencode `internal/llm/tools/write.go`

新内容与现有文件完全相同时返回错误，避免无意义写入、多余权限弹窗和虚假「已修改」状态。
